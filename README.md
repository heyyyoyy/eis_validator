# eis_validator

A small full-stack app for working with **EIS transport package** XML: a **Rust (Axum)** API validates XML against the bundled XSD and extracts payloads from SOAP envelopes, and a **React (Vite)** UI talks to that API through a dev proxy or Nginx in production.

## What it does

- **`GET /health`** — Liveness check with JSON status and timestamp.
- **`POST /validate`** — Upload an XML file (`multipart/form-data`, field `file`). Response indicates whether the document is valid against `schemas/DP_PAKET_EIS_01_00.xsd` and lists XSD errors when invalid.
- **`POST /parse`** — Upload an EIS package XML (Windows-1251 SOAP envelope). Extracts Base64 payloads from `Документ/Контент` and `Прилож/Контент`, decodes them, and returns pretty-printed UTF-8 XML for both.

The backend serves **plain HTTP** only; TLS is provided by Nginx in the production Docker setup.

## Prerequisites

### Run the API locally (without Docker)

- **Rust** (stable) and **Cargo**
- **libxml2** and **pkg-config** (for XSD validation via the `libxml` crate)

```bash
# macOS
brew install libxml2 pkgconf
```

On macOS, if the linker cannot find libxml2, point pkg-config at Homebrew’s keg (adjust the version segment if needed):

```bash
export PKG_CONFIG_PATH="/opt/homebrew/opt/libxml2/lib/pkgconfig"
```

```bash
# Debian/Ubuntu
sudo apt install libxml2-dev pkg-config
```

### Run the frontend locally (without Docker)

- **Node.js** 22+ (or compatible) and **npm**

### Run with Docker

- **Docker** with Compose v2

## Configuration

Copy the example env file and edit as needed:

```bash
cp .env.example .env
```

Variables such as `PORT`, `LOG_LEVEL`, and (for production) `HTTP_PORT`, `HTTPS_PORT`, and cert settings are documented in `.env.example`.

## How to run

### Option 1: Docker Compose — development (recommended for full stack)

`docker-compose.dev.yml` runs the **Vite dev server** (hot reload) and exposes:

| Service  | URL / port |
|----------|------------|
| Frontend | http://localhost:5173 |
| Backend  | http://localhost:3000 |

```bash
docker compose -f docker-compose.dev.yml up --build
```

Open http://localhost:5173 in the browser. API requests to `/health`, `/parse`, and `/validate` are proxied to the backend.

### Option 2: Docker Compose — production (HTTPS)

Builds the static frontend, runs Nginx with TLS (certs generated by a one-shot service), and maps host ports (defaults **80** and **443**):

```bash
docker compose -f docker-compose.prod.yml up --build -d
```

Override `HTTP_PORT`, `HTTPS_PORT`, and cert-related variables in `.env` as needed. The frontend health check hits HTTPS on the container.

### Option 3: Backend only (`cargo`)

From the repository root:

```bash
cargo run
```

Defaults: `HOST=0.0.0.0`, `PORT=3000`, `LOG_LEVEL=info`. Override for a quick local run:

```bash
HOST=127.0.0.1 PORT=8080 LOG_LEVEL=debug cargo run
```

Smoke test:

```bash
curl http://127.0.0.1:3000/health
curl -F "file=@path/to/file.xml" http://127.0.0.1:3000/validate
```

### Option 4: Frontend dev server + local API

Terminal 1 — backend:

```bash
cargo run
```

Terminal 2 — frontend (proxies to http://localhost:3000 by default):

```bash
cd frontend
npm ci
npm run dev
```

Open the URL Vite prints (typically http://localhost:5173). To point the proxy at another backend URL, set `VITE_PROXY_TARGET` (see `frontend/vite.config.ts`).
